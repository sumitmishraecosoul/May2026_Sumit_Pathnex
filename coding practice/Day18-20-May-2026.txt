Today’s focus was GitOps using ArgoCD on Kubernetes. The main idea we repeated in
class: Git is the source of truth, and ArgoCD makes sure the cluster matches Git.


1) What is GitOps (why companies like it)
-----------------------------------------
GitOps means:
- Desired state (k8s yaml / helm) lives in Git
- Any change is done via Git commit
- A controller (ArgoCD) continuously reconciles cluster state to match Git

Why teams prefer it:
- Easy audit trail (who changed what and when)
- Rollback is simple (go back to old Git revision / app history)
- Drift detection (someone changes live cluster manually → Argo shows OutOfSync)
- Repeatable deployments across dev/uat/prod


2) ArgoCD core concepts (what we saw in UI)
-------------------------------------------
Application:
- Defines: repo URL, path, revision, destination cluster + namespace, sync policy

Sync status:
- Synced / OutOfSync / Unknown

Health status:
- Healthy / Progressing / Degraded / Missing

Sync:
- Manual sync = I click/CLI to apply latest Git desired state
- Auto sync = Argo auto-applies when Git changes

Self-heal:
- If someone changes a resource manually, Argo brings it back to Git state

Prune:
- If a resource is removed from Git, Argo deletes it from cluster


3) ArgoCD architecture (high level)
-----------------------------------
- API server: UI/CLI/API entry
- Repo server: clones Git repos and renders manifests
- Application controller: compares desired vs live and syncs
- Dex (optional): SSO/auth

7) Create first app (guestbook) using Application yaml
------------------------------------------------------
guestbook-app.yaml:

apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: guestbook
  namespace: argocd
spec:
  project: default
  source:
    repoURL: https://github.com/argoproj/argocd-example-apps
    targetRevision: HEAD
    path: guestbook
  destination:
    server: https://kubernetes.default.svc
    namespace: default
  syncPolicy:
    automated:
      selfHeal: true
      prune: true

Apply:
kubectl apply -f guestbook-app.yaml

Check:
kubectl get applications -n argocd
kubectl get pods -n default


8) ArgoCD CLI basics (what I practiced)
---------------------------------------
# login (after port-forward)
argocd login localhost:8080

argocd app list
argocd app get guestbook

# manual sync
argocd app sync guestbook

# wait until healthy
argocd app wait guestbook --health

# history + rollback
argocd app history guestbook
argocd app rollback guestbook 1

# delete app (cleanup)
argocd app delete guestbook


9) Practical troubleshooting checklist (job style)
--------------------------------------------------
When app shows OutOfSync:
- Check what drifted: Argo UI diff or argocd app get
- If change was manual and wrong → sync to restore Git state
- If change is required → commit to Git and let Argo sync

When app is Degraded:
- kubectl get pods -n <ns>
- kubectl describe pod <pod>
- kubectl logs <pod>
- check service selectors/ports and deployment readiness probes

Common gotcha from class:
- If auto-sync is ON, rollback may be blocked/overridden; disable auto-sync before rollback.
