Volumes (data that survives container delete)
---------------------------------------------
By default container filesystem is ephemeral. **Volumes** keep data.

  docker run -d \
    -v /path/on/host/html:/usr/share/nginx/html \
    -p 8080:80 \
  --name nginx-vol nginx

  docker volume create mydata
  docker volume ls
  docker volume inspect mydata

I mounted a host folder into nginx and changed index.html from the host — page
updated inside container without rebuild.


Networking basics
-----------------
Containers are isolated; networks control who talks to whom.

Default types from class:
  **bridge**  — default; containers on same bridge can reach each other by name (with compose)
  **host**      — container uses host network stack directly
  **overlay**   — multi-host (Swarm / clustering)
  **none**      — no network

Port publish (host → container):
  docker run -d -p 8080:80 nginx
  # host port 8080 → container port 80

  docker network ls
  docker network inspect bridge


Docker Compose (multi-container app)
------------------------------------
yaml file docker-compose.yml from class:

  version: '3'
  services:
    web:
      image: nginx
      ports:
        - "8080:80"
    db:
      image: mysql
      environment:
        MYSQL_ROOT_PASSWORD: pathnex

Commands:
  docker compose up -d
  docker compose ps
  docker compose logs web
  docker compose down


Docker Registry (share images)
------------------------------
  docker login
  docker tag nginx-image mydockerhubuser/nginx-image:v1
  docker push mydockerhubuser/nginx-image:v1
  docker pull mydockerhubuser/nginx-image:v1

Docker Hub = public registry; companies often use private ECR/Artifactory too.


Swarm vs Kubernetes (class overview)
------------------------------------
  **Docker Swarm** — Docker’s own cluster orchestration (simpler, less common now)
  **Kubernetes** — industry standard for orchestration at scale; nodes run containers
  via container runtime (often containerd). Docker CLI knowledge still helps because
  images/Dockerfile are the same building blocks.

Interview-style answers (from class)
------------------------------------
Q: What is Docker?
A: Platform to build, ship, and run apps in containers — portable, isolated, fast.

Q: Image vs container?
A: Image is template; container is running instance of that image.

Q: Dockerfile vs docker-compose?
A: Dockerfile builds one image; compose runs multi-container stack from YAML.

Q: How is container different from VM?
A: Container shares host kernel, lighter; VM has full guest OS per machine.

Q: How do you debug a failing container?
A: docker logs, docker exec -it, docker inspect, check exit code and port mapping.
