Helm lifecycle commands (class + lab)
-------------------------------------
Upgrade:
  helm upgrade my-nginx-ingress stable/nginx-ingress
  helm upgrade my-nginx bitnami/nginx -n pathnex --set replicaCount=2

Rollback:
  helm history my-nginx -n pathnex
  helm rollback my-nginx-ingress 2
  helm rollback my-nginx 1 -n pathnex

Uninstall:
  helm uninstall my-nginx-ingress
  helm uninstall my-nginx -n pathnex
  helm uninstall my-nginx -n pathnex --keep-history

Dry-run / render without apply:
  helm install my-nginx bitnami/nginx --dry-run --debug
  helm upgrade my-nginx bitnami/nginx -n pathnex --dry-run
  helm template my-nginx bitnami/nginx -f values.yaml

Custom values file (class nginx-ingress example)
------------------------------------------------
Created values.yaml:
  controller:
    replicaCount: 2
    ingressClass: nginx
  service:
    externalTrafficPolicy: Local

Install with override:
  helm install my-nginx-ingress stable/nginx-ingress -f values.yaml


Multi-environment (Pathnex pattern)
-----------------------------------
  helm upgrade --install pathnex-app ./pathnex-chart -f values-dev.yaml -n dev
  helm upgrade --install pathnex-app ./pathnex-chart -f values-staging.yaml -n staging
  helm upgrade --install pathnex-app ./pathnex-chart -f values-production.yaml -n prod

values-dev.yaml (snippet from class):
  replicaCount: 2
  image:
    tag: dev-latest
  service:
    type: ClusterIP
    port: 8080

values-production.yaml (snippet):

  image:
    tag: prod-latest
  service:
    type: LoadBalancer
    port: 80

Interview / mistakes (class)
----------------------------
Q: Helm vs raw kubectl apply?
A: Helm templates + tracks release history + easy rollback.

Q: What is a release?
A: Named instance of a chart running in the cluster.

Mistakes:
  - kubectl edit deployment but Helm owns it → next upgrade reverts
  - secrets in values.yaml committed to Git
  - using :latest tag in production
